===============
-0- definitions
===============
awaitable:
	something that can be awaited
coroutine:
	A function that returns an awaitable which calls
	the body when awaited.
	basically, an `async def`ed function OR the return value
	of an `async def`ed function.
task:
	An awaitable that WRAPS A COROUTINE to run the
	coroutine concurrently (like a thread)
	Tasks are futures.
	methods:
		cancel(msg=None)
		canceled(): was it canceled?
		done(): is it done?
		result(): result of task, only call after awaited
		exception(): the exception when running task
			result() and exception():
				If canceled, raises CancelledError
				If not done, InvalidStateError
		add_done_callback(callback, context=None)
		remove_done_callback(callback)
		get_coro()
		get_name()
		set_name()
futures: "An object representing value of await coroutine."
	?This is basically just any awaitable that has a return value?
	use future.result(timeout=None) to get the result...
	don't see how this is any different from just calling await on a
	task...
NOTE:
	tasks should always have references saved, otherwise they may
	disappear mid-execution (loop only holds weak references)
event loop: the loop that runs awaitables
	await calls in each awaitable are like pause-points and allow
	switching to a different task to allow it to complete or something

context:
	a contextvars.ContextVar instance.
	(like threading.local(), except for asyncio)

cancelling:
	A task is cancelled by raising asyncio.CancelledError
	CancelledError is a BaseException, NOT AN EXCEPTION.
	similar to SystemExit, won't be caught by `except Exception:`


asyncio repl:
	Similar to `python` repl, except asyncio:
	`python -m asyncio`

	============
	-0.1- syntax
	============
	asyncio adds 2 special syntaxes:

	1. async def funcname(args):
	2. await awaitable_expr
	3. async with
	4. async for

	async def commands return an awaitable

	================
	-0.2- interrupts
	================
	ctrl+C might destroy asyncio internals before 3.11.
	after 3.11,
	signal.signal(signal.SIGINT, x)
		where x cancels the main task on the first interrupt
		raises keyboard interrupt on the 2nd interrupt.
			(useful if currently inside a tight loop without any await)

-1- high-level api

	=============
	-1.1- runners
	=============
	Runners are basically wrapped asyncio.EventLoop
	Only 1 asyncio.run() call per thread is allowed.

	asyncio.run(coro, debug=None, loop_factory=None):
	=================================================
		create a new loop to run coro and then close it when done.
		coro: the awaitable to run
		debug: True/False to force, otherwise use system settings
		loop_factory: (3.12+) callable to return a loop replacing "policy" system

	asyncio.Runner(debug=None, loop_factory=None)
	=============================================
		asyncio.run() as a context manager.  Use this to run multiple
		awaitables SERIALLY before closing the loop.

		run(awaitable, context=None)
		============================
			If coroutine, wrapped in a Task. BUT... the task seems to be
			immediately awaited so THEY WILL BE RUN SERIALLY.

		close()
		=======
			* finalize asynchronous generators
			* shutdown default executer
			* close event loop
			* release context
		get_loop()
		==========
			get the EventLoop

		with asyncio.Runner(...) as runner:
			(3.11+)
			runner.run(coro)

	-1.2- coroutines/tasks
	======================
		=============
		-1.2.1- tasks
		=============
		asyncio.Task(coro, loop=None, name=None, context=None, eager_start=False)



		asyncio.create_task(coroutine, *, name=None(3.8), context=None(3.11), eager_start=None(3.14), **kwargs):
			Create a task AND schedule to run.
			`await task` to wait til finished
			task.cancel() to cancel the task
				(use try/finally for proper cleanup)
			NOTE:
				reference to the returned task must be maintained
				or it might be garbage collected.

		===================
		-1.2.2- multitasking
		===================
		Aysncio offers various ways to handle multiple tasks:

			asyncio.TaskGroup: group of tasks
			=================================
				(3.11+)
				Tasks are run when reaching __exit__
				async with asyncio.TaskGroup() as tg:
					t1 = tg.create_task(...)
					t2 = tg.create_task(...)
				All tasks are run together and stopped together on any exception.
					(besides being cancelled)

				methods
				=======
				cancel()
				uncancel()
				add_done_callback(callback):
					callback will be called with task as argument at the end.

			asyncio.gather(aws, return_exc=bool)
			====================================
				similar to:
					with asyncio.TaskGroup() as tg:
						for item in aws:
							tg.create_task(item)
				return_exc:
					return exception in place of return value.
					Otherwise, immediately raise to whoever is awaiting on gather()
					OTHER TASKS CONTINUE RUNNING unlike taskgroup
				cancelled gather cancels all tasks (3.7+)
				cancelled subtask DOES NOT cancel other tasks.
				BUT if return_exc is False, the gather WILL RAISE THE CancelledError
					AND CANNOT BE AWAITED ON AGAIN (
						it'll just immediately reraise the cancelled error)

		==============
		-1.2.3- shield
		==============
		asyncio.shield(task):
			main purpose of shield is to allow cancelling await.
			Use it in place of the actual task.  It doesn't actually
			do anything to prevent the task from being cancelled.

		================
		-1.2.4- timeouts
		================
		Timeouts are implemented as cancelling and raising
		asyncio.TimeoutError

		asyncio.timeout(delay) (3.11+)
			try:
				async with asyncio.timeout(delay):
					dostuff
			except asyncio.TimeoutError
				handle_timeout

		asyncio.timeout_at(when) (3.11+)
			same as asyncio.timeout(time.time()+delay)

		asyncio.wait_for(aw, timeout)
			CANCEL TASK IF TIMEOUT and raise TimeoutError after
			task is cancelled.

		asyncio.wait(awseq, timeout=None, return_when=ALL_COMPLETED)
			done, pending = asyncio.wait(myawaitables)
			must be a sequence.  On timeout, DOES NOT cancel anything

			return_when:
				ALL_COMPLETED
				FIRST_COMPLETED
				FIRST_EXCEPTION

		asyncio.as_completed(aws, timeout=None)
			similar to asyncio.wait(), but return items as soon as
			items are completed.

		===============
		-1.2.5- threads
		===============
		asyncio.to_thread(func, args, kwargs)
			run in a thread, same GIL performance considerations
			apply as with threading.Thread
			wait await for the function to complete.

		asyncio.run_coroutine_threadsafe(coro, loop)
			(3.5.1+)
			Run the coro in the given loop, CALLED FROM ANOTHER THREAD
			Return a concurrent.futures.Future

		asyncio.iscoroutine(obj) 3.4+
