==============
-1- ssh client
==============
ssh target command arg arg arg...

target:
	[user@]host
	ssh://[user@]host:port

Commands
========
NOTE: args will be appended to command separated by a space. This means that
the remote server RECEIVES A SINGLE STRING.  The string itself (or args)
must contain any necessary quoting.
Example1:
	command: ssh target printf '"%s"\n' a b 'hello world'
	result: anbnhellonworldn
	explanation:
		After joining the strings, what the remote would see is:
		printf "%s"\n a b hello world
		Because \n would just become n, and hello world does not contain
		any quoting, the net result would be:
		printf %sn a b hello world

	Proper quoting (single str): ssh target 'printf "\"%s\"\\n" a b "hello world"'
	Proper quoting (multi str): ssh target printf \''"%s"\n'\' a b \''hello world'\'

Alternatively, just redirect stdin
	ssh target <script
	ssh target "bash -s" <script
	ssh target "bash -s" < <(printf '%q ' command line here)

Permissions
===========
Permissions for relevant configuration files should be AT LEAST:
chmod 700 ~/.ssh/
chmod 644 public_key
chmod 644 ~/.ssh/config
chmod 600 private_key
chmod 600 authorized_keys

------
syntax
------
	sequence of key value[s] pairs separated by whitespace
	# comment
	" a single argument with spaces"
	Indentation is ignored.
	Key value pairs apply to the most recent HOST patterns line.
	For each option, the first match applies.  More specific options
	should be at the top and general defaults at the bottom.
	keys are case INSENSTIVE

------------
useful keys:
------------
	Host: patterns separated by whitespace (ORed together)

	HostName: the actual host name to use (ip,
		www.somewhere.com, etc)

	IdentitiesOnly: [yes|no] only use the IdentityFile even if
		ssh agent gives other identities

	IdentityFile: specify the private key file
		The file name may use the tilde syntax to refer to a
		user's home directory or one of the following escape
		characters:
			'%d' local user's home directory
			'%u' local user name
			'%l' local host name
			'%h' remote host name
			'%r' remote user name
		Multiple files are allowed (tried in order)

	LocalForward: port forwarding from local machine to remote machine.
		2 arguments: [<bindaddr>:]port <hostname at remote>:port
		put ipv6 addresses in [] or use <ipv6addr>/port
		<bindaddr>: which interface to bind to. localhost = only local can use
			empty or * = all interfaces

	PasswordAuthentication [yes|no]

	Port: port to connect to at remote

	RemoteForward: forward remote port to local same format as
		LocalForward, except order is <remote addr> <local addr>
		empty means loopback only

	ServerAliveInterval: secs, send a keepalive message to
		server asking for a response.

	TCPKeepAlive: [yes|no] send TCP keepalive messages, will dc if
		network is down temporarily

	User: specify the user to ssh as

---------
patterns:
---------
	*: 0 or more chars
	?: 1 character
	!: negate the match
	can have commma delimited for a "pattern-list"
	(an AND)

For each option, the EARLIEST MATCH is used.
These configs would be equivalent for
hostnames `name`, `tame`, and `aim`
config1:
	HOST name
		USER myuser
	HOST *e
		USER bad
		SERVERALIVEINTERVAL 69
	HOST *
		IDENTITIESONLY yes

config2:
	HOST name
		USER myuser
		SERVERALIVEINTERVAL 69
		IDENTITIESONLY yes
	HOST tame
		USER bad
		SERVERALIVEINTERVAL 69
		IDENTITIESONLY yes
	HOST aim

================
-1.3- forwarding
================
ssh can perform port forwarding via ssh tunneling
adds ssh security to the connection... (at least for
the segment between the 2 machines)

Forward from client to ssh host
	ssh -L [<localhost>]:<localport>:<remotehost>:<remoteport>
		bind a socket to <localhost>:<localport> which forwards data
		to <remotehost>:<remoteport>

Forward from ssh host to client
	ssh -R [remotehost]:<remoteport>:<localhost>:<localport>
		ssh to the remote machine, and create a socket bound to
			<remotehost>:<remoteport> which forward traffic to
			<localhost>:<localport>

These would force you to create an ssh session and leave it open.
This is fine if you will be using the ssh session as well.
Alternatively, add the -f option and the command sleep 10 or
something similar.  This will cause the tunnel to open for the
specified sleep duration.  During this time, connect over the tunnel.
When the sleep finishes, the ssh will exit and the tunnel will last
as long as the connection lasts.

=====================
-3.2- authorized_keys
=====================
These files are on the ssh server and allow that particular user to login
using public/private key pairs.

	~/.ssh/authorized_keys
	~/.ssh/authorized_keys2

lines STARTING with # are comments.
Each line should be a space-separated sequence of:
	[options...] keytype b64publickey comment
-------
options
-------
	from="pattern-list"
		comma-separated list of patterns that host must be from.
	permitlisten="[host:]port,[host:]port..."
	permitopen="[host:]port,[host:]port..."
		Explicitly specify which host/port is allowed
		listen=-R, open=-L
		ipv6 must be in square brackets
	restrict
		Pack these options together:
			no-port-forwarding
			no-agent-forwarding
			no-X11-forwarding
			no-pty
			no-user-rc
	[no-]agent-forwarding
		specifies whether agent forwarding is allowed
	[no-]port-forwarding
		specify whether port forwarding is allowed
	[no-]pty
		Specify whether to give a pty
	[no-]user-rc
		Specify whether ~/.ssh/rc should be used
	[no-]X11-forwarding
		whether X11 forwarding is allowed
	cert-authority
		Specify that this is a CA to validate users.
	command="command"
		Use this command, ignore any user commands
	environment="name=value"
		Specify an environment variable.  PermitUserEnvironment must be on.  Repeatable
	expiry-time="timespec"
		This key is only accepted before "timespec"
	no-touch-required
		only for FIDO authenticator algs (sk-*)
	verify-required
		only for FIDO authenticator algs (sk-*)
	tunnel="n"
		Force a tun(4) device on server otherwise next available device
		if requests a tunnel

--------
keytypes
--------
	sk-ecdsa-sha2-nistp256@openssh.com
	ecdsa-sha2-nistp256
	ecdsa-sha2-nistp384
	ecdsa-sha2-nistp521
	sk-ssh-ed25519@openssh.com
	ssh-ed25519
	ssh-dss
	ssh-rsa

NOTE: this means that this usually matches the format of *.pub file when generating a key.
so in general, you can just cat (append) into the corresponding authorized_keys file

====================
-4.1.2- certificates
====================
Certificates consist of
* public key
* identity info
* 1 or more principal names (user/host)
* options signed by a CA key

Certificates can have different formats:
	OpenSSH
	X.509

----------------
Creating a cert:
----------------
ssh-keygen [-h] -s /path/to/ca_key -I key_id \
	[-n user1,user2...|host1,host2...] \
	[-V start:end] \
	/path/to/key.pub
result: /path/to/key-cert.pub

-------
options
-------

	clear
	critical:name=contents
	extension:name=contents
	force-command=command
	[no|permit]-agent-forwarding

	[no|permit]-user-rc
	[no|permit]-x11-forwarding
	no-touch-required
	source-address=addrlist
	verify-required
