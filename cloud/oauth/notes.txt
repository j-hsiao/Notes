===========
-0- general
===========
OAuth is a method for user to login and give an app access to their resources.
The general steps are:
1. user requests resource/site login
2. Send user to authorization server
3. user logs in
4. User gets redirected to a redirection uri which the app usually controls
5. The redirection uri gets the authorization code
6. Use the code to get access token

Generally, access tokens are used via the Authorization header
Authorization: Bearer token_goes_here

===========
-0.1- terms
===========
authorization server
	the server that users log into
authorization code
	a number (short lived) that can be used to get a token.
access token
	a medium-lived string used as a key for access to resources
refresh token
	a longer-lived string used to get more access tokens before expiration.
redirect_uri
	After authorization, the server will redirect the browser to this uri.  In
	general, this can be localhost ?in which case any port is allowed even if
	not explicitly configured?  If NOT localhost, then port must be fixed.

===============
-1.1- auth code
===============
https://accounts.google.com/o/oauth2/auth

==================
-1.1.1- parameters
==================
These are provided as query strings:

client_id: the client id from registering the app
redirect_uri: must exactly match a configured uri
	NOTE that url is readable by any third-party scripts, so the redirect.  This
	means the authorization would be visible by anything also displayed on the
	redirect uri.  The redirect uri should thus not contain anything.
response_type: code
scope: space-delimited list of scopes
access_type: online(default)|offline
	offline = get an access token AND a refresh token (can get access even if user is afk)
state: some state for the app, recommend adding a random string
	for verification, reduce Cross Site Request Forgery:
	???  a phishing site "guesses" some kind of url to perform an action on your
	server if a user has cookies that have authorization to your server visits the
	phishing site in some way, cookies get sent to server too which then seems
	legit...???
include_granted_scopes: if true, then new token also has any previously-granted scopes
login_hint: email or sub identifier
prompt: [none|consent|select_account], if omitted, only prompt on first request.

These ones only appear under the desktop app section:
code_challenge: urlsafe_b64encode_unpadded(sha256(code_verifier))
code_challenge_method: plain|S256 (case sensitive!)

================
-1.1.2- response
================
The response values are as a query string added after the redirect_uri

error: the error code
code: the auth code
iss: https://accounts.google.com
scope: https://www.googleapis.com/auth/drive.file, whatever scope was in the initial request?

==================
-1.2- access token
==================
POST https://oauth2.googleapis.com/token
Content-Type: application/x-www-form-urlencoded

==================
-1.2.1- parameters
==================
client_id: from registering app
client_secret: from registering app
	NOTE: docs say this is "optional", but in practice it seems like it is required
	even if it is not used, regardless of the app type? (both installed and web)
code: from the query param
grant_type: must be `authorization_code`
redirect_uri: any of the configured urls
code_verifier: urlsafe_base64_unpadded of a string (probably uuid)

optional DPoP header

================
-1.2.2- response
================
response is application/json
success:
	access_token
	expires_in
	refresh_token
	refresh_token_expires_in
	scope
	token_type
error
	error
	error_description

=============
-1.3- refresh
=============
Same as -1.2.1-, except instead of code/redirect_uri,
give `refresh_token` query string.

============
-1.4- revoke
============
POST https://oauth2.googleapis.com/revoke?token=the_token

revokes a token and corresponding refresh token, removing any granted scopes.


=============
-2- microsoft
=============
https://learn.microsoft.com/en-us/entra/identity-platform/v2-oauth2-auth-code-flow

===============
-2.1- auth code
===============
https://login.microsoftonline.com/{tenant}/oauth2[/v2.0]/authorize

==================
-2.1.1- parameters
==================
Parameters are provided via query string to the authorization uri.

______
tenant
	required; the tenant to use: [common|organizations|consumers|tennant ids]
	This is the first part of the endpoint in the
	login.microsoft.com/{tenant}/... url
	Generally use `common`
_________
client_id
	required; the id for the registered application
	can use d50ca740-c83f-4d1b-b616-12c519384f0c (abraunegg's onedrive app id...)
_____________
response_type
	required; space-delimited string of [code|id_token|token]
____________
redirect_uri
	required; The redirect uri to send info to. It MUST have been registered and
	match exactly to a registered uri.
	typical options:
		https://login.microsoftonline.com/common/oauth2/nativeclient
		http://localhost
	It seems like after logging in from the browser, the user will be redirected
	to this `redirect_uri` with extra query params containing the token or
	something maybe...  Normally, the redirect_uri would be a link that the
	application controls.  When the application receives a GET http request, it
	knows that an authentication result is in the query strings. (or fragment
	or post, depending on response_mode)
_____
scope
	required; space-separated list of scopes. This seems to be X.permission
	examples:
		Files.ReadWrite[.All]
		Sites.ReadWrite[.All]
		offline_access
		openid (required to get an id token)
		profile
		email
_____________
response_mode
	recommended; The method to return response information.
	query: return as ?name=value&name=value... query string, default if only getting code
	fragment: return as #name=value&name=value..., default if getting id token
	form_post: use POST method, ?generally the most recommended?

	NOTE: fragment ?is safer? because it does not get sent to any servers, it
	remains in the browser only.
______
prompt
	optional
	type of user interaction:
	login: force login
	none: try using single-login, return "interaction_required" error if fail
	consent: triggers permission dialog
	select_account: choose an account
__________
login_hint
	optional; prespecify user/email, usually from `login_hint` JWT claim of previous sign-in
___________
domain_hint
	optional; similar to login_hint, but use `tid` JWT claim of previous sign-in
______________
code_challenge
	recommended; Client creates a "code_verifier" and sends a hash or some
	transformation.  The server records the final result and the method.  Client
	gets an access token, sending the original code_verifier as proof that the
	client is the original source of the request (server transforms code_verifier
	using the given method and it should match `code_challenge`)
_____________________
code_challenge_method
	recommended; plain(default)|S256, method to encode `code_verifier`
_____
state
	recommended; a value returned in the token response to prevent cross-site
	request forgery attacks In the response, it should be identical to the initial
	request.
_____
nonce
	required for an id token, inside the returned `id_token` as claim in JWT

================
-2.1.2- response
================
code=authorization_code
id_token=token
	contains a special `c_hash` JWT claim which is a hash of `code`.
state=state_from_parameters

===================
-2.1.3- error codes
===================
error=code
	invalid_request
	unauthorized_client
	access_denied
	unsupported_response_type
	server_error
	temporarily_unavailable
	invalid_resource
	login_required
	interaction_required
error_description=desc

==================
-2.2- access token
==================
POST to login.microsoftonline.com/{tenant}/oauth2[/v2.0]/token
==================
-2.2.1- parameters
==================
Parameters are given as application/x-www-form-urlencoded in the POST body.

extra parameters:
____
code
	the authorization code.
__________
grant_type
	[authorization_code]
_____________
code_verifier
	The original of the `code_challenge` from getting the auth code.
_____________
client_secret
	Only for confidential apps.

_____________________
client_assertion_type
	'urn:ietf:params:oauth:client-assertion-type:jwt-bearer' to use a cert
________________
client_assertion
	JWT, signed with the registered cert for app-only mode.

================
-2.2.2- response
================
Response is json
____________
access_token
	used for api
__________
token_type
	`Bearer` only
__________
expires_in
	duration til expiration in seconds
_____
scope
	nonstandard, default value is to use whatever the authorization code says.
_____________
refresh_token
	only if `offline_access` scope is requested.
________
id_token
	only if `openid` scope was requested.

==============
-2.2.3- errors
==============
Errors are returned in json as well, same format as response, but with
different keys.

error
	an error code
	invalid_request
	invalid_grant
	unauthorized_client
	invalid_client
	unsupported_grant_type
	invalid_resource
	interaction_required
	temporarily_unavailable
	consent_required
	invalid_scope
error_description
	description
error_codes
	list of STS-specific error codes for debugging
timestamp
	for debugging
trace_id
	for debugging
correlation_id
	for debugging

=============
-2.3- refresh
=============
refresh requests are identical to getting access token with minor differences:
1. use `refresh_token` instead of `code`
2. no redirect_uri
3. grant_type=`refresh_token`
4. no code_verifier
